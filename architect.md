Ниже — **Markdown-архитектура** под твой кейс: минимальная, офлайн-ориентированная, с `PWA / локальный HTML + xlsx + Web Worker`, и с добавленными пунктами для **Tauri**. `SheetJS` удобно читать/собирать Excel в браузере, а для тяжёлой обработки и экспорта можно уводить расчёты в Web Worker и собирать файл как `Uint8Array`/Blob для скачивания. [jstool.gitlab](https://jstool.gitlab.io/demo/sheetjs-xlsx-js/)

## 1. Цель приложения

- Принять на вход **2 файла** формата `.xls/.xlsx/.csv`.
- Распарсить их по фиксированной структуре.
- Свести данные по правилам.
- Выдать итоговый `.xlsx` или `.csv`.
- Работать быстро на больших таблицах и без сервера.

## 2. Рекомендуемый стек

- UI: `PWA` или просто `local HTML + JS`.
- Парсинг/экспорт: `xlsx` (`SheetJS`).
- Тяжёлая обработка: `Web Worker`.
- Хранение настроек: `localStorage` или `IndexedDB`.
- Для упаковки desktop-версии: `Tauri` как отдельная ветка.

`PWA` хорошо подходит для офлайн-сценария, потому что Service Worker кэширует shell приложения и позволяет работать без сети. [samsunginternet.github](https://samsunginternet.github.io/pwa-series-service-workers-the-basics-about-offline/)
`Tauri` даёт более лёгкую desktop-оболочку и при этом ограничивает доступ к файловой системе через scoped permissions, что удобно для безопасного локального инструмента. [viadreams](https://viadreams.cc/es/blog/tauri-guide/)

## 3. Логика обработки

1. Пользователь загружает два файла.
2. Главный поток валидирует типы и отправляет файлы в worker.
3. Worker читает файлы через `SheetJS`.
4. Worker приводит данные к единой схеме.
5. Worker выполняет сведение/агрегацию.
6. Worker генерирует итоговый `xlsx` или `csv`.
7. UI получает результат и предлагает скачать файл.

`SheetJS` поддерживает чтение/парсинг Excel в браузере, а результат можно конвертировать в HTML, массивы или файл для выгрузки. [sheetjs](https://sheetjs.com/demos/manifest)
Для генерации файла из worker обычно формируют `Uint8Array`, а затем в UI создают `Blob` и скачивание через `download`. [github](https://github.com/SheetJS/sheetjs/issues/2743)

## 4. Структура проекта

```md
project/
├─ index.html
├─ app.js
├─ worker.js
├─ styles.css
├─ manifest.webmanifest
├─ sw.js
├─ lib/
│  └─ xlsx.full.min.js
└─ config/
   └─ mapping.json
```

### Назначение файлов
- `index.html` — интерфейс загрузки и кнопка запуска.
- `app.js` — основная логика UI и обмен с worker.
- `worker.js` — чтение, обработка и генерация результата.
- `sw.js` — офлайн-кэш для PWA.
- `mapping.json` — схема сопоставления колонок и правил.

## 5. Поток данных

```md
Input file A  ┐
              ├─> app.js ─> Web Worker ─> merged result ─> app.js ─> download
Input file B  ┘
```

## 6. Почему так

- **Легковесность**: без сервера, без базы, без лишнего UI-фреймворка.
- **Скорость**: worker не блокирует интерфейс.
- **Переносимость**: открывается как обычный сайт или как локальная страница.
- **Простота поддержки**: один сценарий, одна схема входных данных.

## 7. PWA-слой

- Добавить `manifest.webmanifest`.
- Добавить `sw.js` для кэширования HTML, JS, CSS и библиотеки `xlsx`.
- Делать работу offline-first.
- Хранить пользовательские настройки локально.

Service Worker обычно кэширует app shell и отдает его офлайн, а обновление ресурсов контролируется через `install`, `fetch` и `activate` события. [bitsofco](https://bitsofco.de/bitsofcode-pwa-part-1-offline-first-with-service-worker/)

## 8. Tauri-дополнения

### Когда Tauri полезен
- Нужен **настоящий desktop app** без браузерной оболочки.
- Нужен доступ к локальной файловой системе с контролем прав.
- Нужен стабильный запуск “как программа”, а не как вкладка браузера.

### Что добавить в архитектуру
- `src-tauri/` как отдельный backend-слой.
- Команды чтения/записи файлов через Tauri API.
- Scoped permissions для доступа только к нужным папкам или файлам.
- Вынести тяжёлые операции в Rust только если JS станет узким местом.

Tauri использует более строгую модель доступа к файловой системе через permissions/capabilities, что удобно для локальной утилиты с ограниченным набором операций. [evilmartians](https://evilmartians.com/chronicles/making-desktop-apps-with-revved-up-potential-rust-tauri-sidecar)

### Когда Tauri не нужен
- Если приложение всегда запускается из браузера.
- Если тебе достаточно file input + download.
- Если не нужен системный доступ к папкам.

## 9. Практический выбор

- **Сейчас**: `local HTML + xlsx + Web Worker + PWA`.
- **Потом, если захочется desktop**: перенос в `Tauri`.
- **Electron**: только если критичны готовые desktop-паттерны и ты готов платить весом приложения.

## 10. Итоговая рекомендация

Для твоего кейса я бы делал именно **PWA/локальный HTML**, потому что это даёт минимум веса, максимум контроля и проще всего поддерживается.  
`Tauri` стоит оставить как следующий этап, если захочется упаковки в desktop-формат без тяжести Electron. [electronjs](https://www.electronjs.org/docs/latest/tutorial/performance)

Могу следующим сообщением оформить это уже как **готовый технический план по файлам**, например:
- `index.html`
- `app.js`
- `worker.js`
- `sw.js`
- `manifest.webmanifest`

или сразу дать **стартовый шаблон кода** под эту архитектуру.
